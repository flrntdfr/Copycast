# Filesystem as the only persistence layer

Copycast deliberately has no database — not even an embedded one. All state lives in the data directory: one directory per Mirror containing yt-dlp's download archive (dedup), the audio files, their `.info.json` sidecars (episode metadata), the last successfully fetched original feed XML for RSS Sources (channel metadata), and one small descriptor file holding the Source URL and Mirror settings. The directory tree is fully self-describing: human-readable, rsync/backup-friendly, and restorable with no migrations. At the scale of a personal podcast archive, a database (embedded SQLite/H2 was the considered alternative) would add schema and migration overhead while duplicating state that yt-dlp already tracks on disk — and a divergence between DB and filesystem would be a new failure mode that plain files cannot have.
